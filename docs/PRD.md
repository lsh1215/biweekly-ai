# PRD: Warehouse Picker — Agentic VLA 물류 피킹 로봇

> 최종 업데이트: 2026-03-29
> 상태: Draft

---

## 1. 프로젝트 요약

### 1.1 한줄 소개

LLM 기반 에이전틱 오케스트레이터가 주문을 해석하고 계획을 세우며, VLA 모델이 로봇 팔을 제어하여 창고 선반에서 물건을 피킹하는 시뮬레이션 시스템.

### 1.2 핵심 가치

- **Hierarchical VLA 아키텍처**의 실제 구현 (Planner-Executor-Verifier)
- **산업용 로보틱스** 도메인 — 물류/이커머스 업계와 직결
- **오류 복구 루프** — 피킹 실패, 물체 가림, 그립 미끄러짐 등 현실 변수 대응
- **업계 표준 기술 스택** — ROS2 + Gazebo + Python

### 1.3 목표 사용자

포트폴리오 프로젝트로서의 타겟 오디언스:
- 로보틱스/물류 자동화 기업 채용 담당자
- AI/ML 엔지니어 포지션 면접관
- 로보틱스 연구실

## 2. 문제 정의

### 2.1 현재 문제

물류 창고에서 주문 처리 시:
1. 주문서에 여러 물건이 포함됨 (멀티 아이템)
2. 물건들이 서로 다른 선반에 분산되어 있음
3. 피킹 순서에 따라 효율이 크게 달라짐
4. 물건이 가려져 있거나 예상 위치에 없을 수 있음
5. 그립 실패, 충돌, 물체 낙하 등 물리적 변수 존재

### 2.2 현재 VLA 모델의 한계

- 단일 pick-and-place는 가능하지만 멀티스텝 주문 처리는 불가
- 피킹 순서 최적화(경로 계획)를 자체적으로 수행 못함
- 예상치 못한 상황(물건 가림, 그립 실패)에 대한 자율 대응 불가
- 주문서 파싱, 재고 확인 등 인지적 작업 불가

### 2.3 솔루션

에이전틱 오케스트레이터(Claude)가 "두뇌" 역할을 하여:
- 주문서를 파싱하고 피킹 계획을 세움
- VLA에게 단일 피킹 태스크를 순차적으로 하달
- 카메라 피드백으로 성공/실패를 판단
- 실패 시 원인을 추론하고 대안 계획을 수립

## 3. 기능 요구사항

### 3.1 MVP (Phase 1) — 기본 피킹 루프

**목표**: 단일 주문의 순차 피킹 + 오류 복구

#### F1: 주문 처리 시스템

| ID | 요구사항 | 우선순위 |
|---|---|---|
| F1.1 | 자연어 주문 입력 ("주문 #1234: 사과 1개, 물병 1개, 책 1권") | P0 |
| F1.2 | Planner가 주문을 파싱하여 피킹 리스트 생성 | P0 |
| F1.3 | 각 아이템의 선반 위치 매핑 (선반 ID → 3D 좌표) | P0 |
| F1.4 | 피킹 순서 결정 (가까운 것부터 순차) | P1 |

#### F2: 로봇 피킹 실행

| ID | 요구사항 | 우선순위 |
|---|---|---|
| F2.1 | VLA가 자연어 명령("선반 A3에서 사과를 집어")을 받아 로봇 액션 생성 | P0 |
| F2.2 | 로봇 팔이 목표 물체에 접근 → 그립 → 들어올리기 → 박스에 놓기 | P0 |
| F2.3 | 그리퍼 개폐 제어 (물체 크기에 따른 그립 힘 조절) | P1 |
| F2.4 | 피킹 완료 후 홈 포지션 복귀 | P0 |

#### F3: 검증 및 오류 복구

| ID | 요구사항 | 우선순위 |
|---|---|---|
| F3.1 | Verifier가 카메라로 피킹 성공 여부 판단 | P0 |
| F3.2 | 그립 실패 감지 → Planner에 보고 → 재시도 | P0 |
| F3.3 | 물체 낙하 감지 → 바닥에서 재피킹 계획 수립 | P0 |
| F3.4 | 물체가 보이지 않을 때 → "앞의 물체 치우기" 재계획 | P1 |
| F3.5 | 최대 재시도 횟수 (3회) 초과 시 해당 아이템 스킵 + 보고 | P0 |

#### F4: 시뮬레이션 환경

| ID | 요구사항 | 우선순위 |
|---|---|---|
| F4.1 | Gazebo 창고 환경 (선반 3~5개, 각 선반에 물체 2~4개) | P0 |
| F4.2 | 로봇 팔 1대 (6-DOF, 그리퍼 포함) | P0 |
| F4.3 | RGB-D 카메라 (오버헤드 + 손목) | P0 |
| F4.4 | 물체 물리 시뮬레이션 (중력, 마찰, 충돌) | P0 |
| F4.5 | 피킹 결과를 담는 박스/컨테이너 | P0 |

### 3.2 Enhanced (Phase 2) — 경로 최적화 + 멀티 주문

## 4. 비기능 요구사항

### 4.1 성능

| 지표 | 목표값 | 비고 |
|---|---|---|
| 단일 피킹 성공률 | ≥ 80% | VLA 단독 대비 +10% 이상 |
| 멀티스텝 주문 완료율 (3아이템) | ≥ 60% | 오류 복구 포함 |
| Planner 응답 시간 | < 5초 | `claude -p` 기준 |
| VLA 추론 시간 | < 500ms/액션 | SmolVLA on MPS |
| 시뮬레이션 프레임레이트 | ≥ 30 FPS | Gazebo on Docker |

### 4.2 안정성

- 시뮬레이션 크래시 시 자동 재시작
- Planner 타임아웃 시 기본 동작(재시도) 실행
- 로그 전체 기록 (주문 → 계획 → 실행 → 결과)

### 4.3 확장성

- VLA 모델 교체 가능 (SmolVLA → OpenVLA → π0)
- Planner 교체 가능 (Claude → GPT-4o → 로컬 LLM)
- 시뮬레이터 교체 가능 (Gazebo → Isaac Sim)
- 새로운 물체/선반 추가 시 코드 변경 최소화

### 4.4 코드 품질

- Python 타입 힌트 전체 적용
- ROS2 노드 단위 테스트
- 에이전트 로직 단위 테스트 (Planner 응답 모킹)
- Docker Compose로 원클릭 실행

### 5.2 데이터 흐름

```
1. User → "주문 #1234: 사과 1개, 물병 1개"
2. Orchestrator → Planner (claude -p):
   Input:  "주문: 사과(A3), 물병(B2). 현재 로봇 위치: 원점. 피킹 계획 세워줘"
   Output: {"tasks": [{"item":"사과","shelf":"A3","action":"pick"}, ...]}
3. Orchestrator → Task Manager: 첫 번째 태스크 실행
4. Executor → VLA (SmolVLA): "선반 A3에서 사과를 집어"
   Output: [joint_angles: [0.1, -0.3, 0.7, ...], gripper: 0.8]
5. Executor → Gazebo: Joint Command 발행
6. Gazebo → Camera: RGB-D 이미지 피드백
7. Orchestrator → Verifier (claude -p):
   Input:  [카메라 이미지] + "사과를 성공적으로 집었는가?"
   Output: {"success": false, "reason": "사과가 바닥에 떨어져 있음"}
8. Orchestrator → Planner: 재계획
   Output: {"tasks": [{"item":"사과","location":"floor","action":"pick_from_floor"}]}
9. 반복...
```

### 5.3 상태 머신

```
IDLE → PLANNING → EXECUTING → VERIFYING → SUCCESS
                      │            │
                      │            ▼
                      │        REPLANNING ─→ EXECUTING (재시도)
                      │            │
                      │            ▼ (재시도 초과)
                      │         SKIPPED
                      │
                      ▼ (모든 아이템 완료)
                   COMPLETED
```

| 상태 | 설명 |
|---|---|
| IDLE | 주문 대기 |
| PLANNING | Planner가 피킹 계획 수립 중 |
| EXECUTING | VLA가 로봇 액션 실행 중 |
| VERIFYING | Verifier가 결과 확인 중 |
| REPLANNING | 실패 → Planner가 대안 계획 수립 |
| SUCCESS | 현재 아이템 피킹 성공 → 다음 아이템으로 |
| SKIPPED | 최대 재시도 초과 → 해당 아이템 스킵 |
| COMPLETED | 주문의 모든 아이템 처리 완료 |

## 6. 기술 스택 요약

| 레이어 | 기술 | 비고 |
|---|---|---|
| **Planner** | `claude -p` (구독 기반) | Max 구독, API 키 불필요 |
| **Verifier** | `claude -p` + 이미지 입력 | 비전 기반 성공/실패 판단 |
| **Executor** | SmolVLA (450M) + PyTorch MPS | 맥북 로컬 추론 |
| **Middleware** | ROS2 Jazzy | 표준 로보틱스 미들웨어 |
| **Simulator** | Gazebo Harmonic | 업계 표준 시뮬레이터 |
| **Container** | Docker (osrf/ros:jazzy-desktop) | 맥북에서 실행 |
| **Language** | Python 3.11+ | 전체 통일 |
| **ML Framework** | PyTorch + LeRobot | SmolVLA 로딩/추론 |

|---|---|---|---|

| Gazebo Docker가 맥북에서 느림 | 중 | 중 | 물리 스텝 간격 조절, 환경 단순화 |
| Claude -p 레이턴시가 루프에 병목 | 중 | 낮음 | 비동기 호출, 캐싱, 배치 계획 |
| ROS2 + Gazebo 학습 곡선 | 중 | 높음 | 공식 튜토리얼 우선, 점진적 복잡도 |
| VLA 모델이 도메인 특화 데이터 없이 일반화 실패 | 높음 | 중 | 시뮬레이션 내 데이터 수집 → 파인튜닝 (Phase 2) |
