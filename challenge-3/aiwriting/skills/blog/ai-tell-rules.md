# AI 티 제거 규칙 (R1-R7)

ai-tell-scrubber subagent 가 시작 시 로드. 결정론적 grep 게이트로 종료 조건 검증.

## 핵심 명제

사람이 쓴 글과 AI 가 쓴 글의 결정적 차이는 **보편성**. AI 는 누구에게나 맞는 문장을 쓴다. 사람은 자기만의 상황·실패·수치를 쓴다.

---

## R1. 의인화 (anthropomorphism)

- **R1.a** 헤딩(h1/h2) hook 의인화는 글당 **최대 1회** 허용. 본문이 hook 을 분석으로 이어가는 경우 한정.
- **R1.b** 본문 의인화는 **0회**.
- **R1.c** wordplay 보존 - 동음이의·반복어를 활용한 의도된 수사는 보존 (예: "결과는 깔끔했다. 깔끔하게 전부 실패했다.", "살아있는 채로 잡혀있다.").

## R2. 드라마/평가어

- **R2.a** `증발` 류 드라마 동사는 본문 0회. 사실 동사로 치환.
- **R2.b** `악질` 류 평가어는 본문 0회. 헤딩 hook 슬롯에서 R1.a 와 합산 1회만 허용.

## R3. 메타 클로징

- **R3.a** `이게 ~다`, `한 일은 그뿐이다`, `이것이 ~의 전부다`, `**결론.**` - 전부 삭제.
- **R3.b** 삭제 자리에 `## 요약` 섹션 삽입.

## R4. Thesis 진입부

- **R4.a** 글 첫머리의 `>` 인용구 thesis - 삭제. 구체적 Problem 으로 바로 시작.
- **R4.b** 글 첫머리의 `**Thesis.**` 굵은체 단락 - 라벨만 제거하지 말고, **첫 문장을 메커니즘 단언으로 재작성**. 단순 삭제 시 진입 임팩트 손실.
  - Before: `**Thesis.** 지연은 실패로 다루고, 빨리 잘라내야 시스템이 숨을 쉰다.`
  - After: `지연은 실패와 다르게 호출 스레드를 천천히 점유한다. 자원이 소진될 때까지.`

## R5. 헤딩 hook 일관성

한 코퍼스(블로그 시리즈) 내에서 같은 등급의 hook 은 같은 처리. 시리즈의 한 글에서 hook 을 보존했으면 다른 글의 동급 hook 도 보존.

## R6. 요약 섹션

- **R6.a** 다음 챕터 예고 금지: `Phase X 에서`, `다음 글에서`, `곧`, `이후 다룬다` 0회.
- **R6.b** 사실 단언과 메커니즘 압축으로만 구성: 검증 수치 + 작동 원리 + 한계.
- **R6.c** 미완료 작업 암시 0회. "X 측에서도 짝이 필요" → "X 측 보호도 같은 글의 Trade-offs 에서 다뤘다" 같은 사실 회수형.

## R7. 타이포그래피 - em-dash · en-dash 금지

한국어 기술 블로그에서 em-dash(`—`, U+2014)와 en-dash(`–`, U+2013)는 **AI 가 쓴 글이라는 명백한 시그널**. 한국어 모국어 작성자는 자판에 없는 이 문자를 거의 쓰지 않습니다. ChatGPT/Claude 가 영문 타이포그래피 습관을 한국어에 그대로 옮겨 찍는 자국입니다.

- **R7.a** em-dash(`—`, U+2014) 본문 · 헤딩 · 전부 0회.
- **R7.b** en-dash(`–`, U+2013) 본문 · 헤딩 · 전부 0회.
- **R7.c** **1차 대체는 hyphen-minus(`-`, U+002D)**. em-dash 가 자리잡았던 자리에 부호가 필요하면 기본적으로 hyphen-minus 로 받습니다.
- **R7.d** **부호의 의미가 분명히 다른 자리면 그 부호로**:
  - 항목 나열의 도입: 콜론 `:`
  - 인과 · 흐름 · 계층 관계: 화살표 `→` 또는 `>`
  - 독립된 두 문장 사이: 마침표로 분리
  - 동격 · 열거 묶음: 중간점 `·`
- **R7.e** 코드 블록 안의 `--` 옵션 (`--guards on`), URL 의 hyphen, 마크다운 리스트 마커(`- `), 기술 합성어(`at-least-once`, `request-per-thread`)는 모두 정상 용도. 검사 제외.

### em-dash 교체 가이드

판단 순서: **(1) 콜론 · 화살표 · 마침표 중 하나에 의미가 명확히 맞으면 그것. (2) 아니면 hyphen-minus.** em-dash 자리를 무조건 다른 부호로 바꾸려 하지 말고, default 는 hyphen 으로 받습니다.

| Before (em-dash) | After (적절한 대체) | 선택 이유 |
|---|---|---|
| 헤딩 `# 제목 - 부제목` | `# 제목 - 부제목` | 부제목 구분, hyphen 이 default |
| `Aggregate - 정의, 탄생, 적용` | `Aggregate - 정의, 탄생, 적용` | 단순 구분, hyphen |
| `세 가지 - A, B, C` | `세 가지: A, B, C` | 나열 도입은 콜론이 의미상 정확 |
| `이론은 간단하다 - 구현하면 어렵다` | `이론은 간단하다. 구현하면 어렵다.` | 독립된 두 문장은 마침표로 분리 |
| `핵심은 X 다 - 그게 전부다` | `핵심은 X 다. 그게 전부다.` | 두 단언 강조, 마침표 |
| `Order(Root) - Variant - Image` | `Order(Root) > Variant > Image` | 계층 · 포함 관계는 화살표가 의미 정확 |
| `재고 예약 → 결제 - 실패 시 재고 해제` | `재고 예약 → 결제 - 실패 시 재고 해제` | 화살표가 이미 흐름이라 다음 부호는 hyphen |

원칙: em-dash 가 하던 역할의 절반은 한국어에서 hyphen 으로 자연스럽게 받힙니다. 콜론 · 화살표 · 마침표는 부호의 의미가 명확히 다른 경우에만.

## 금기 표현 카탈로그 (한국어)

| 금기 | 이유 | 교체 |
|---|---|---|
| `~을 통하여`, `~함으로써` | 영어식 번역체 | `~으로`, `~해서` |
| `~에 대해`, `~에 관해서` | 공허한 연결 | 대상 직접 지칭 |
| `매우`, `굉장히`, `정말로`, `실제로` | 빈 수식어 | 수치로 대체 또는 삭제 |
| `따라서`, `그러므로`, `또한`, `게다가` 남발 | AI 문장 연결 패턴 | 문단 구조로 |
| `다양한 ~`, `여러 ~`, `많은 ~` | 구체성 없음 | 숫자 |
| `제공합니다`, `존재합니다`, `활용하여` | 번역체 | `있다`, `한다`, `써서` |
| `~할 수 있었습니다` 남발 | 수동적 서술 | `~했다` |
| `~것 같습니다` | 모호한 완곡 | 단정형 또는 근거 |
| `시너지`, `선도적인`, `글로벌한` | 비즈니스 클리셰 | 구체적 사실 |
| `열정적으로`, `최선을 다해`, `꾸준히` | 추상 수식어 | 수치 · 행동 |
| `—` (em-dash, U+2014) | **AI 명백한 시그널** | 1차 hyphen `-`, 의미 따라 `:`, `→`, `.` (R7) |
| `–` (en-dash, U+2013) | **AI 명백한 시그널** | 위와 동일 (R7) |

## 본문 의인화 카탈로그

| Before | After |
|---|---|
| Payment 는 여전히 죽어 있다. Order 는 신경 쓰지 않는다 | Payment 는 여전히 기동되지 않았다. Order 는 영향을 받지 않는다 |
| 헬스체크까지 죽었다 | 헬스체크까지 응답을 멈췄다 |
| 시스템이 숨을 쉽니다 / 쉰다 | (의인화 - 메커니즘 사실로 재작성) |
| 시스템에게도 사용자에게도 더 정중한지 | (감성 의인화 - 삭제) |
| "X" 와 "Y" 가 운명을 공유하면서 | "X" 와 "Y" 가 같은 트랜잭션에 묶여서 |
| 무관한 API 도 함께 죽는다 | 무관한 API 도 함께 응답을 멈춘다 |

| Before | After |

## 종료 조건 (결정론적 grep 게이트)

다음 grep 결과가 **모두** 충족돼야 scrubber 가 종료.

### 삭제 검증 (전부 0건)

```
grep "이게 [A-Za-z가-힣]* 다\|한 일은 그뿐\|이것이 [A-Za-z가-힣 ]*의 전부\|^\*\*결론\.\*\*" → 0
grep "증발" → 0
grep "신경 쓰지 않는다\|운명을 공유\|숨을 쉰다\|숨을 쉽니다\|더 정중한지" → 0
grep "악질인 이유" → 0
grep "^> [^|]" (글 첫 10줄 내 thesis 인용구) → 0
grep "^\*\*Thesis\.\*\*" → 0
```

### [R7] em-dash · en-dash 검증 (코드 블록 외부 0건)

```
grep "—" → 0   (em-dash, U+2014)
grep "–" → 0   (en-dash, U+2013)
```

코드 블록(``` ... ```) 안의 `—`/`–` 는 외부 라이브러리 · 로그 출력에 포함될 수 있으므로 보존. 검증은 본문(non-code) 만 대상.

자동 치환 가이드 (scrubber 가 적용, 우선순위 순):

1. 의미가 명확히 콜론 · 화살표 · 마침표 중 하나에 맞으면 그것 (위 R7 가이드 표 참고).
2. 그 외 모든 자리: `\s—\s` → `\s-\s` (양쪽 공백 hyphen-minus 로).
3. 헤딩 `# X - Y` 의 default 는 `# X - Y`.
4. 단독 `—` (공백 없이) 는 문맥 보고 `,` 또는 `-`.

### 보존 검증 (anti-grep, 의도된 표현은 1건 이상 유지)

각 글마다 어떤 표현을 보존했는지 변경 리포트에 명시. wordplay 와 헤딩 hook 은 보존된 채로 grep 1건 이상 카운트되어야 한다.

### 요약 섹션 검증

```
grep "^## 요약" → 1
글의 ## 요약 섹션 내부에서:
  grep "Phase\s*[0-9]\|다음 글\|곧 .\|예정$\|이후 [가-힣]" → 0
```

### 균일 호흡 검증 (heuristic)

3개 이상 연속 문장이 비슷한 길이·종결 (`~다.` 만 4번 이상 연속, `~입니다.` 만 4번 이상 연속) 인 단락이 있다면 **하나는 짧게 또는 종결 변주** 권고. 강제 변경 X, 리포트에만 표시.

## 출력 리포트 형식 (필수)

scrubber 는 종료 시 다음 형식 리포트 반환:

```
## AI 티 제거 리포트

### 적용된 변경 (N 건)
| Rule | Before | After | 줄 |
|---|---|---|---|
| R1.b | 신경 쓰지 않는다 | 영향을 받지 않는다 | L110 |
| ... | ... | ... | ... |

### 보존된 표현 (anti-grep, M 건)
- L1 헤딩: "Payment 하나가 죽었을 뿐인데" (R1.a)
- L7: "결과는 깔끔했다" (R1.c)
- ...

### 검증 결과
- 삭제 grep: ALL 0건 ✓
- 보존 anti-grep: M 건 ✓
- ## 요약 섹션: 1건 ✓
- 미래 시제: 0건 ✓
- 균일 호흡 경고: K 단락 (수정 권고만)

### 종료 상태
PASS / NEEDS_HUMAN_REVIEW [균일 호흡만 남은 경우]
```
